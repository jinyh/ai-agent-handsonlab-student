## 二、外部 Context：你主动给 Agent 的材料

### 2.1 文件上传 vs 文本粘贴 vs 链接提供

| 方式 | 适用场景 | 优点 | 缺点 | 最佳实践 |
|------|---------|------|------|---------|
| **文件上传** | 财报PDF、合同扫描件、长报告 | 保留原始格式、方便引用页码 | PDF解析可能丢失格式、大文件可能截断 | 先转换为纯文本或Markdown再上传；超过50页的先做摘要 |
| **文本粘贴** | 简短材料、关键段落、数据表格 | 格式可控、Agent处理快 | 手动操作繁琐、长文本易出错 | 先用人工筛选出关键段落；用```代码块包裹表格数据 |
| **链接提供** | 新闻文章、官网页面、公开数据 | 直接引用来源、信息最新 | Agent可能无法访问（需要联网权限）、页面结构复杂时解析不准 | 优先提供可访问的静态页面；对关键页面先手动复制主要内容 |
| **结构化数据** | 财务数据、用户数据、实验数据 | Agent易于计算和对比、输出准确 | 需要预先整理成CSV/JSON/表格 | 用CSV格式最佳； column名要清晰；缺失值标注为N/A |

### 2.2 信息密度管理

**核心原则**：Agent 的上下文窗口有限（通常几万到几十万token），信息不是越多越好。

#### 反模式：信息过载

```
❌ 给 Agent 上传：
- 某公司2020-2024年全部年报（5份，每份200页）
- 10篇行业研究报告
- 50条新闻链接

结果：Agent "迷失"在细节中，输出泛泛而谈，抓不住重点。
```

#### 最佳实践：分层投喂

```
✅ 分层策略：

第一层：摘要（人工预处理）
- 用1页纸总结5份年报的核心变化
- 用表格整理10篇报告的核心结论
- 这页纸给 Agent 作为"导航图"

第二层：详细材料（按需深入）
- 当 Agent 需要核实某个数据时，再提供具体页面的文本
- "请查看2023年年报第45页的营收明细表格"

第三层：原始文件（最后防线）
- 当 Agent 对某项数据有疑问时，回到原始PDF核实
- 人做最终守门
```

### 2.3 Context 质量检查清单

在把材料交给 Agent 前，问自己：

- [ ] **相关性**：这份材料和当前任务直接相关吗？有没有"噪音"？
- [ ] **时效性**：数据是最新的吗？Agent 的训练数据截止到什么时候？
- [ ] **完整性**：有没有遗漏关键信息？（如财报只给了利润表，没给现金流量表）
- [ ] **准确性**：来源可靠吗？有没有明显矛盾的数据？
- [ ] **可读性**：格式是否便于 Agent 解析？（PDF表格→纯文本表格）

## 三、对话 Context：历史对话的累积

### 3.1 上下文窗口的概念

> "Agent 能'记住'你们之前说了什么，但这个记忆是有限的。就像你和下属开了一个很长的会，聊到第20个话题时，他可能已经忘记了第1个话题是什么。"

**关键认知**：
- 不同模型的上下文窗口不同，从几万到几十万 token 不等。
- 窗口越大，不等于越适合塞入更多材料；噪音越多，Agent 越难抓重点。
- 具体窗口大小会随模型版本变化，课堂中以所选 Provider 的官方说明为准。

**这意味着**：
- 如果你的对话和材料超过模型可处理范围，早期或低优先级信息可能被忽略
- 对于复杂任务，20-30轮对话后，Agent 可能开始"失忆"

### 3.2 应对"失忆"的策略

#### 策略1：定期重置核心目标

```
用户：请帮我分析宁德时代的财报。（第1轮）
...
（经过15轮讨论，话题已经漂移到了电池技术路线）

用户：回到我们的核心任务——基于2024年Q3财报，评估投资价值。
请忽略之前关于技术路线的讨论，重新聚焦于财务指标分析。
```

> "这不是 Agent 的bug，而是它的设计特性。在长对话中，定期'重置'核心目标，是保持输出质量的关键。"

#### 策略2：关键信息反复提及

```
用户：请记住，我们的核心假设是"产能过剩是短期现象"。
基于这个假设，请重新评估之前提到的3个风险因素。
```

> "重要信息不要只提一次。在关键节点重复核心假设和约束条件，可以对抗'失忆'。"

#### 策略3：拆分长对话

```
会话1：信息收集（让 Agent 读取材料、提取数据）
→ 保存输出

会话2：分析框架（让 Agent 设计分析框架、确定对比维度）
→ 保存输出

会话3：洞察生成（基于前两个会话的输出，生成最终结论）
→ 保存输出
```

> "对于特别复杂的任务，不要试图在一个会话中完成所有事情。拆分成多个会话，每个会话聚焦一个阶段。"

#### 策略4：使用系统 Context（见下节）

把核心规则写入项目规则文件，这样即使对话变长，Agent 也会"记得"自己的身份和约束。OpenCode 常用 `AGENTS.md`，Claude Code 常用 `CLAUDE.md`。

## 四、系统 Context：项目规则文件

### 4.1 什么是系统 Context

系统 Context 是通过配置文件定义的"持久化记忆"。你把规则写进文件，Agent 在每次对话开始时都会读取这些规则。

**类比**：
> "就像公司的《员工手册》。新入职时发一次，但里面的规则长期有效。你不需要每天重复'上班要打卡'，因为手册里已经写了。"

### 4.2 Claude Code 与 OpenCode 的规则文件

不同终端 Agent 使用的规则文件名不同，但本质相同：都是给 Agent 的长期岗位说明书。

| 工具 | 常用规则文件 | 教学说法 |
|------|--------------|----------|
| Claude Code | `CLAUDE.md` | 本课推荐基线的项目规则文件 |
| OpenCode | `AGENTS.md` | 合规备用方案的项目规则文件 |
| 其他 Agent | 可能有不同文件名 | 关注规则内容，不死记文件名 |

#### 示例1：金融投研版 AGENTS.md（适合 S1-S4）

```markdown
# AGENTS.md

## 角色
你是某投研/战略分析团队的资深分析师。

## 工作原则
- 只基于给定材料分析
- 不确定时标注[待核实]
- 每个关键结论必须有材料依据
- 不编造公司数据、政策细节或市场份额

## 输出规范
- 优先使用 Markdown 表格
- 先给结论，再给依据
- 风险和不确定性单独列出
- 涉及数据时标注来源或[待核实]

## 禁忌事项
- 不给买入、卖出、持有、目标价等投资建议
- 不使用没有来源的精确数字
- 不把推测写成事实
- 不忽略反面证据

## 校验机制
- 输出最后列出最不确定的 2 个点
- 输出最后列出需要人工复核的 2 个数据
- 如果材料不足，明确说"信息不足"，不要猜测
```

#### 示例2：产业咨询版 AGENTS.md（适合 S5-S8）

```markdown
# AGENTS.md

## 角色
你是某管理咨询公司的资深顾问。

## 输出规范
- 优先使用 Markdown 表格和框架图
- 先给洞察，再给支撑依据
- 优势和劣势、机会和风险必须平衡呈现
- 涉及数据时标注来源或[推断]

## 禁忌事项
- 不给具体的商业决策建议（如"应该进入这个市场"）

### 4.3 系统 Context 的最佳实践

| 实践 | 说明 | 示例 |
|------|------|------|
| **定义角色** | 让 Agent "知道"自己是谁 | "你是资深分析师" vs "你是实习生"，输出质量差异巨大 |
| **明确风格** | 量化风格指标 | "每条洞察不超过40字"比"简洁"更可执行 |
| **列出禁忌** | 明确告诉 Agent 不能做什么 | "禁止编造数据"比"请准确"更有效 |
| **内置校验** | 让 Agent 自我检查 | "完成后列出3个不确定点" |
| **解释术语** | 确保 Agent 理解你的行业黑话 | 在文件中定义常用缩写和术语 |
| **版本管理** | 项目规则文件也需要迭代 | 根据实战效果不断调整，保留历史版本 |

### 4.4 系统 Context 的局限性

- **不是万能药**：项目规则文件只能覆盖"通用规则"，具体任务的细节仍需在 Prompt 中说明
- **需要更新**：当项目阶段变化时（如从"研究阶段"进入"汇报阶段"），需要更新项目规则文件
- **不能替代人工**：项目规则文件减少了重复说明，但最终的输出质量仍取决于人的校验

## 五、工具 Context：MCP 与 Skills

### 5.1 MCP（Model Context Protocol）

MCP 是 Anthropic 推出的开放标准，让 Agent 可以安全地访问外部工具和数据源。

**类比**：
> "MCP 就像你给下属配的团队。没有 MCP，他只能靠自己查资料；有了 MCP，他可以调用数据团队、法务团队、市场团队的能力。"

#### 常用 MCP Server

| MCP Server | 功能 | 适用场景 |
|-----------|------|---------|
| **filesystem** | 访问本地文件系统 | 读取/写入项目文件、遍历目录 |
| **github** | 访问 GitHub API | 读取代码仓库、提交issue、查看PR |
| **postgres** | 访问 PostgreSQL 数据库 | 查询结构化数据、生成报表 |
| **brave-search** | 网络搜索 | 实时搜索最新信息 |
| **puppeteer** | 浏览器自动化 | 抓取网页内容、截图 |

#### 课堂演示边界

课堂当天只要求理解 MCP 的作用，不要求所有小组现场完成配置。

建议讲师只演示 1 个预配置 MCP：

| 演示选项 | 推荐度 | 说明 |
|----------|--------|------|
| filesystem / 本地资料访问 | 高 | 最直观：Agent 能读取指定目录材料 |
| 搜索 / 网页抓取 | 中 | 价值明显，但网络和 API 稳定性要求更高 |
| GitHub / 数据库 / 浏览器自动化 | 低 | 技术味较重，容易偏离 MBA 课堂目标 |

课堂话术：

> "MCP 可以理解为给 Agent 接外部工具。没有 MCP，Agent 主要靠你给它材料；有 MCP，它可以按规则访问文件、网页、数据库等外部系统。但今天大家只需要理解这个概念，不需要现场配置。"

### 5.2 Skills（技能）

Skills 是预定义的工具使用流程。你可以把一套常用的 Agent 操作打包成一个"技能"，以后一键调用。

**类比**：
> "Skills 就像 Excel 的宏。你把一套常用的操作录下来，以后点一下按钮就能执行。"

#### Skills 的获取方式

1. **使用工具内置或官方目录**：不同 Agent 的 Skills/插件生态不同
2. **从 GitHub 或 ZIP 导入**：适合进阶学员和讲师准备课堂模板
3. **自定义编写**：把常用流程写成可复用配置

#### 工作坊中推荐的 Skills

课堂中不要求安装 Skills，而是用"Skill 卡片"帮助学员理解：Skills 就是把常用 Prompt + Context + Harness 封装成企业 SOP。

#### 示例：财报快速解读 Skill 卡片

```markdown
# Skill：财报快速解读

适用场景：上市公司季报/年报初筛

输入：
- 财报 PDF 或文本节选
- 关注指标清单
- 同业对比对象

执行步骤：
1. 提取核心指标
2. 标注数据来源
3. 识别积极变化和风险
4. 列出不确定点
5. 输出投委会摘要

禁止：
- 不给买入/卖出/持有建议
- 不编造缺失数据
```

其他可设计的 Skill 卡片：

| Skill 名称 | 功能 | 适用场景 |
|-----------|------|---------|
| **市场调研速览** | 搜集公开信息、生成对比表格 | 市场调研、竞品分析 |
| **会议纪要整理** | 提取结论、争议点和 Action Items | 会议管理、项目跟进 |
| **商业模式画布** | 基于材料填充 BMC 九宫格 | 战略分析、商业计划 |

1. ...
2. ...
3. ...

## 禁止事项

- 不把假设写成事实

3. 各方立场
4. Action Items
5. 风险或争议点

## 边界
- 不补充没有出现在会议中的信息
- 说话人不明时标注[说话人不明]
- 模糊表达保留原意并标注[需确认]
```

## 七、Context 裁剪方法：从一堆材料到一页导航

面对大量材料时，不建议直接全部交给 Agent。先做一页"导航 Context"，告诉 Agent 每份材料的作用。

### 裁剪步骤

| 步骤 | 动作 | 产物 |
|------|------|------|
| 1 | 列材料目录 | 材料编号、标题、来源、日期 |
| 2 | 标重点段落 | 每份材料 1-3 个关键段落 |
| 3 | 分事实/观点/假设 | 避免 Agent 混淆材料与用户判断 |
| 4 | 标注用途 | 告诉 Agent 哪份材料回答哪个问题 |
| 5 | 明确缺口 | 主动告诉 Agent 哪些信息没有 |

### 示例：裁剪前后

裁剪前：

```text
我上传了 8 份资料，请帮我写一份行业分析。
```

裁剪后：

```text
你将看到 4 份材料：
- A：政策原文节选，用于判断合规要求
- B：行业报告摘要，用于判断市场结构
- C：两家企业官网摘录，用于判断产品能力
- D：客户访谈纪要，用于判断真实痛点

请先不要写报告。第一步只输出"材料事实表"：
材料编号 | 可确认事实 | 可能推断 | 信息缺口 | 适合用于报告哪一节
```

讲师提示：

> "一页导航 Context 的价值，是让 Agent 先知道地图，再开始走路。"

## 八、事实、假设、问题三分法

非技术学员最容易犯的错误，是把自己的行业判断混在材料里交给 Agent。建议使用三分法：

| 类型 | 定义 | 写法 | Agent 应该怎么处理 |
|------|------|------|--------------------|
| 事实 | 材料中明确出现的信息 | "材料 A 提到..." | 可作为结论依据 |
| 假设 | 用户目前的判断或猜测 | "我们的初步假设是..." | 需要找支持和反证 |
| 问题 | 需要 Agent 帮忙回答的内容 | "请判断..." | 输出分析或信息缺口 |

可复制模板：

```text
请注意区分以下三类信息：

【事实】
- ...

【我的假设】
- ...

【希望你回答的问题】
- ...

要求：
1. 事实可以作为依据。
2. 假设不能直接当作结论。
3. 如果材料不足以回答问题，请写[信息不足]。
```

## 九、Context 管理的实战原则

### 原则1：质量 > 数量

```
❌ 给 Agent 100页未筛选的材料
✅ 给 Agent 5页人工整理的关键摘要 + 50页原始材料作为"备用查询库"
```

### 原则2：结构化 > 自由文本

```
❌ "这是某公司的信息：他们在做电池，客户有特斯拉，收入挺高的..."
✅ 
```
公司：宁德时代
主营业务：动力电池、储能电池
核心客户：特斯拉、宝马、奔驰、蔚来
2024年Q3营收：XXX亿元（同比+XX%）
```
```

### 原则3：可追溯 > 可信

```
❌ 要求 Agent "确保准确"
✅ 要求 Agent "标注每条数据的来源，不确定时标注[待核实]"
```

### 原则4：分层 > 一步到位

```
❌ 在一个会话中完成：读材料 → 做分析 → 写报告 → 做PPT
✅ 拆分为4个会话，每个会话保存输出，作为下一个会话的输入
```

### 原则5：系统 + 具体 = 最佳

```
项目规则文件（系统级）：定义角色、风格、禁忌、校验机制
  +
Prompt（任务级）：定义本次任务的具体目标、Context、格式
  =
高质量输出
```

## 十、Context 管理检查清单

在启动 Agent 任务前，检查：

- [ ] **外部 Context**：我提供的材料是否相关、准确、结构化？
- [ ] **对话 Context**：这个任务是否需要拆分成多个会话？
- [ ] **系统 Context**：项目规则文件是否已定义角色、风格、禁忌？
- [ ] **工具 Context**：需要哪些 MCP/Skills？是否已配置？
- [ ] **信息密度**：材料是否经过人工筛选？有没有信息过载？
- [ ] **时效性**：数据是否最新？Agent 的训练数据是否有盲区？
- [ ] **可追溯**：是否要求 Agent 标注数据来源？
- [ ] **事实/假设分离**：是否明确告诉 Agent 哪些是材料事实，哪些是我的判断？
- [ ] **缺口显性化**：是否要求 Agent 标注材料没有覆盖的问题？
- [ ] **阶段归档**：长任务是否保存了中间产物，便于开启新会话继续？

## 十一、一句话总结

> **管理 Context 的本质，是管理 Agent 的"视野"和"记忆"。给它看对的东西、以对的格式、在对的时机，它的输出质量就会显著提升。**
