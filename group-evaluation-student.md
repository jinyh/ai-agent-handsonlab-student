# 小组评价说明（学员版）

> **用途**：帮助各组理解下午实战的成果要求、汇报结构和评分维度。
> **核心原则**：评分不是目的，目的是看清楚什么是好的 Agent 协作。

---

## 一、每组最终需要展示什么

每组汇报时需展示：

1. **最终成果**：Agent 生成的最终输出，例如报告、表格、纪要、任务分析。
2. **Prompt 迭代记录**：至少展示两个版本的 Prompt，并说明为什么调整。
3. **翻车记录**：Agent 犯过什么错误？你们怎么发现？怎么纠正？
4. **心得总结**：用 1 句话总结今天最大的收获。

建议每组指定一名记录员，负责记录 Prompt 版本、Agent 输出问题和人工校验过程。

---

## 二、评分维度与权重

| 维度 | 权重 | 评价要点 |
|------|------|----------|
| **Prompt 设计** | 25% | 目标清晰、结构完整、受众明确、格式具体、约束到位 |
| **Context 管理** | 25% | 材料相关、经过筛选、格式合适、信息不过载 |
| **Harness 约束** | 20% | 有边界约束、防幻觉机制、输出格式和校验机制 |
| **输出质量** | 20% | 成果可用、无明显幻觉、逻辑清晰、关键数据经核实 |
| **迭代过程** | 10% | 记录了问题、调整原因和至少一次有效改进 |

---

## 三、每个维度怎么准备

### Prompt 设计

检查自己的 Prompt 是否回答了这些问题：

- [ ] 要完成什么具体任务？
- [ ] 输出给谁看？
- [ ] 用什么风格和语气？
- [ ] 输出成什么格式？
- [ ] 有哪些不能做的事？

### Context 管理

检查提供给 Agent 的材料：

- [ ] 是否与任务直接相关？
- [ ] 是否经过人工筛选？
- [ ] 是否用表格、清单或清晰段落表达？
- [ ] 是否说明了材料中的不确定点？
- [ ] 是否避免一次性塞入过多无关信息？

### Harness 约束

建议至少加入这些约束：

```text
只基于给定材料，不要补充未提供的信息。
材料中没有提及的信息，请标注[材料中未提及]。
不确定的结论，请标注[待核实]。
完成后列出最不确定的 2-3 个点。
```

### 输出质量

人工校验时重点看：

- [ ] 有没有编造数据、公司事实或政策细节？
- [ ] 关键数字是否和材料一致？
- [ ] 输出格式是否符合要求？
- [ ] 结论是否有材料依据？
- [ ] 是否清楚标注了不确定点？

### 迭代过程

建议汇报时展示：

```text
Prompt V1：最初怎么写
问题：Agent 输出哪里不对
调整：我们加了什么 Context / Harness
Prompt V2：修改后版本
结果：输出有什么改善
```

---

## 四、汇报建议结构

每组 5 分钟，建议按下面顺序：

| 时间 | 内容 |
|------|------|
| 30 秒 | 场景与目标 |
| 1 分钟 | 最终版 Prompt |
| 1 分钟 | 迭代过程 |
| 1 分钟 | 翻车与纠正 |
| 1 分钟 | 最终成果 |
| 30 秒 | 一句话总结 |

---

## 五、底线提醒

- 不要把 Agent 输出当成最终事实。
- 不要让 Agent 给出买入、卖出、持有、目标价等投资建议。
- 涉及公司、政策、市场数据时，必须标注来源或不确定性。
- 翻车不是扣分点；没有发现问题、没有校验，才是风险点。
